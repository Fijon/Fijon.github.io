DailyRollingAppender多进程文件覆盖处理

2017-11-04
Java
log4j

多进程下，Java项目使用同一份log4j配置文件，导致多进程往同一份文件写日志，最终在日志文件发生滚动时，造成日志内容缺失。

业务场景

该任务是在处理Mapreduce提交的任务时，通过默认的log4j的DailyRollingFileAppender类进行日志文件滚动备份时，发现数据丢失的问题引起。
原因：
　　1、Hadoop集群中设置了一台机器可以同时启动两个Mapper和两个Reduceer，总共四个jvm进程。由于这四个文件读取的都是同一份日志配置文件，所以当一台机器启动两个或者两个以上的进程时，会导致多个jvm进程的日志写入同一份文件。由于log4j自己本身机制，多个进程同时往一个文件写入，并不会造成文件串行的情况。但是却会发生另外一种情况，即在日志文件发送滚动时，会导致日志内容丢失。
　　2、同样的情况，也发送在storm集群上。除了storm集群上默认收集的work日志，我们还需要新增一个另外的模块日志，来供统计平台收集处理。
解决方案：
　　为了避免这种情况，公司前辈实现了一个写日志文件的方式。简单的在log4j的基础上封装了一些接口，并是不同的jvm进程的日志，记录在不同的文件中。具体：在新增一个日志文件之前，会对该文件进行加锁。确认同一时刻内，只允许一个进程持有该文件。当有新的进程启动，并记录日志的时，先判断是否有进程持有该文件。如果没有则使用往该文件记录日志内容。否则则改写到其他文件。
缺陷：
　因为接口的抽象成都不高，导致日志接口与业务和框架耦合性非常高，丢失了logging日志框架的高通用性。

上边这种处理情况，虽然可以解决日志覆盖的情况，但是处理非常粗糙。存在很多问题，比如日志文件记录的内容不能安装时间排序，无法通过配置的范式配置备份的数量。

分析

　　为了方便按时间查询日志，我们需要按照时间轴来滚动日志。
　　为了模拟线上的情况，我在Windows做了测试，发现日志到了整点滚动时，并不会发生日志覆盖的情况，只是发生无法进行重命名文件，并且所有的日志文件都打在一个文件上，并不会进行日志文件切割。
　　之前猜想，之所以会出现日志覆盖的情况是存在这种情况，A进程和B进程同时对某个文件进行写入操作，导致二者之间出现相互覆盖的问题。从Windows环境测试来看，不会出现这个问题，而且与线上的日志滚动的问题也是不相同。
我将写好的测试代码，放到ubuntu环境重新跑了一下，出现了与线上环境一样的情况。而且该现象仅仅出现在多进程写日志，并且在滚动的时候，才会发生。
查看了log4j的DailyRollingFileAppender的实现。发现代码中使用subAppend（LoggingEvent）方法来判断是否进行日志滚动处理。

当发生日志滚动时，调用rollOver()方法进行日志文件滚动处理。

　　从代码可以看出，滚动时，按照以下逻辑进行判断：
1、判断文件重命名格式是否存在。
2、如果存在，则关闭当前文件（取消该文件占有权，并将内容写入到缓存区中）。
3、判断滚动后的文件名称是否存在，如果存在则进行删除处理，否则采用renameTo方法对当前文件进行重命名。

　　另外，对于renameTo的处理，Linux系统与Windows系统的区别还是挺大的。如果同时又A，B两个进程同时对同一文件f进行操作，并且某一线程优先对文件进行重命名操作，将其重命名为f1。
对于Windows来说，该文件被被其他进程所使用，不允许进行重命名操作。
对于Linux来说，系统持有的是该文件的句柄，而不是直接持有该文件。假设A进程先对f进行重命名操作，那么文件被重命名成功，并且A进程，新建f文件，将日志内容写入到f文件中。而B进程持有的是f文件的句柄，此时该句柄对应的是f1文件。即b文件依旧写入f1文件中。
在日志滚动时，Windows环境下，当A进程对f进行重命名时，因为B进程持有该文件，所以重命名失败。两者同时拥有该文件，谁也不允许其他人修改文件信息。导致不断的往该文件写入内容。最终日志文件会异常庞大。 在Linux环境下，A进程重命名成功，系统上存在f1，当B进程进行重命名时，会将f1文件删除，并将f文件重命名为f1文件。此时f1文件的内容存储的是A进程重命名之后的内容。导致A进程和B进程在前一时间段的日志内容丢失。

修改

　　鉴于rollOver()方法权限是default级别，因此没有办法通过继承的方式进行修改，除非在不修改原有log4j jar包的情况下，新增一个org.apapche.log4j的包名。通过这种方式进行修改进行继承修改。修改主要分为两种方面，一个是修改日志的缓存区（针对Windows环境），一个是修改rollOver()逻辑，如果存在该文件，不进行删除操作，而是直接写入到新的文件中。通过另外一种方式是新增一个类，然后将DailyRollingFileAppender整个类拷贝过去，修改rollingOver()方法。

实际上在处理的时候，还在思考一个多进程问题，加入A，B进程同时进入While循环（A进程对target重命名之前，B进程也进入进入了while循环），此时会否存在A进程重命名之后，B进程再次重命名？
现实处理中是不会存在的。
　　在Windows环境下，假如A进程重命名成功，此时f文件被重命名为f1，文件系统不再存在f文件。接着A进程退出循环，B进程进去重命名f文件。此时f文件不存在，B进程重命名失败，进入下一个循环，由于target文件存在，B进程退出循环。但是存在理论上的一种情况，A进程重命名成功后，退出循环，并且已经新建f文件，并往里边写入信息了，此时B进程在进行重命名，那么是会造成文件丢失的。
Linux环境下，与Windows下环境不同。A，B进程同时拥有f文件的句柄。A进程重命名后，f文件变为f1文件，但是B进程拥有的还是原来f文件的句柄（现在为f1文件句柄），写入的是f1文件。A进程新增f文件，并持有该文件的句柄。B进程对f1文件进行重命名为f1，不影响。
ps:在Windows下，假想情况没有遇到，但是总觉得还是个坑。

Storm集群

　　storm集群比Hadoop集群，改起来比较容易。Storm的日志框架是采用logback，相关的配置文件在storm目录下的logback目录下的cluster文件中。在storm脚本中，启动nimbus, ui, supervisor, core时，都会通过指定参数的方式，携带上文件。通过全文搜索storm项目，在Clojure文件中采用系统设置变量的方式的方式指定相关配置。鉴于storm的worker端口号范围是固定的，则可以根据worker端口来配置worker进程的日志路径。与Mapreduce集群不同的，Mapreduce无法确认job中mapper和reducer之间的共性。

其他方法

　　其实，多进程的日志记录还有另外一种写法，就是使用SocketAppender类，将日志发送到指定的日志服务器上。由于在分布式环境下，如果多个机器往同一台机器上写日志文件的话，会导致该机器的IO资源非常紧缺。如果不往同一台机器上写，那么需要在多台机器上部署多个接收日志内容的服务器，处理起来比较麻烦。所以没有采用该方法。
